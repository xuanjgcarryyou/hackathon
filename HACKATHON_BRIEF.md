## Problem Definition

- **Who hurts:** 有企業午餐訂餐需求的中大型公司 HR / 行政人員，同時需要填寫 ESG / 碳足跡報告，卻沒有數據來源
- **What hurts:** 每次訂外送就產生大量一次性餐具，但沒有系統能追蹤「到底減少了多少」；ESG 報告裡的包材碳排數據要手動估算，耗時且不可信
- **Why now:** 台灣環境部 2026 年起強化減塑政策；企業 ESG 揭露壓力上升（GHG Protocol Scope 3）；Loopick 等案例已驗證循環餐具可行，但缺乏企業端整合平台
- **Evidence of pain:** 現有試辦案例痛點：店家存放空間不足、回收點不普及、消費者主動歸還意願低 → 封閉企業場景（辦公室）可系統性繞過這三個問題

---

## Target User

- **Primary user:** 企業 HR / 行政人員（Admin）—— 負責訂餐安排與 ESG 報告的人
- **User context:** 每週需設定員工午餐方案，每季需向主管或稽核單位提交環保相關數據，目前靠 Excel 手動估算，數據品質差
- **What they currently do instead:** 繼續使用一次性餐具 + 沒有數據 + ESG 報告空白或填假數字

---

## Proposed Solution

- **Product form:** Web 平台（B2B SaaS）+ 手機掃碼（PWA 或 Web App）
- **Core value proposition:** 企業訂午餐的同時，自動追蹤循環容器流向，一鍵生成符合 GHG Protocol Scope 3 格式的 ESG 報表段落
- **Key features for demo:**
  1. **HR 儀表板**：設定本週訂餐（餐廳 + 份數 + 環保服務商），系統顯示預估減少一次性餐具數與 CO₂e
  2. **QR 掃碼收回**：管理員掃描容器批次 QR，記錄回收數量；系統即時更新容器循環率與異常警示
  3. **ESG 報表生成**：點擊「生成報表」→ AI 產出本季度 ESG 段落文字（中/英）與圖表建議，可直接貼入正式報告
- **Out of scope:**
  - B2B 市集（Module A）：Demo 時用固定假資料模擬已合作餐廳與環保服務商
  - 即時 IoT 硬體感測：用 QR 掃碼模擬，不需要 RFID 或 GPS
  - 金流 / 押金系統：Demo 不涉及實際付款
  - 消費者（員工）個人 App：員工端只展示點餐選單，不是主要 Demo 路徑

## Decision

建立一個 **B2B 循環午餐管理 SaaS 平台**，以企業封閉場景（辦公室）為切入點，解決「循環容器無法規模化歸還」的根本障礙。平台連結環保服務商、合作餐廳與企業客戶，讓 HR 能在現有午餐訂餐流程中同步完成容器循環追蹤，並由 AI Agent 自動生成 ESG 報表段落。Demo 聚焦一個完整循環的 Happy Path：HR 設定 → 員工點餐 → 管理員掃碼回收 → ESG 報表輸出。

---

## Rationale

- **選封閉場景（辦公室）而非開放場景（外送平台）：** 開放場景依賴消費者主動歸還，行為不可靠；辦公室場景有管理員、有文化壓力、有固定回收點，不需改變個人行為。
- **選 B2B 而非 B2C：** 一個企業客戶 = 數百個使用者，繞過冷啟動問題；企業有 ESG 報告需求 = 明確的付費動機。
- **選 ESG 報表生成作為差異化鉤子：** 現有循環容器案例（Loopick、PackAge+）都只做物流，沒有做數據到報表的轉換；這是真實企業痛點，且技術上用 Claude API 可在 hackathon 內實現。
- **模組 A Mock 而非全做：** B2B 市集的核心價值在於「可以選」，假資料足以在 Demo 中傳達，節省 10+ 小時開發時間給核心模組。

## Rejected Options

| Option | Why Rejected |
|--------|-------------|
| 逆向物流（外送員回程收容器）| 違反便利性原則（對外送員增加工作量）；外送員是自僱 gig worker，無法要求額外任務 |
| 押金制借杯平台（B2C） | 依賴消費者主動歸還，行為不可靠；冷啟動需要大量商家合作才有價值 |
| 高品質材質 + 現有回收鏈（純材料循環路徑）| 循環價值可見度低，Demo 難以打動評審；缺乏平台層面的差異化 |
| 外送平台默認環保選項（純 C 方向）| 影響力強但屬於政策倡議，缺乏技術深度；Demo 沒有「系統」可以展示 |
| 全面 IoT 容器追蹤（每個杯子 UUID + GPS）| 過度設計，違反 problem-framing-brief §2.2；硬體成本與整合時間超出 hackathon 範圍 |

## Risks

## Fallback Plan

若開發時間不足，Demo 最小可行版本：
1. **HR 儀表板**（靜態頁面 + 假資料，可點擊設定餐廳/份數）
2. **ESG 報表生成**（呼叫 Claude API，輸入預設數字，輸出真實報表文字）
3. **容器循環率數字**（靜態展示，不需要掃碼流程）

這三件事即使沒有後端 API、只有前端 + Claude API 直接呼叫，也能在 30 分鐘內搭出可 Demo 的版本，足以說服評審整個系統的概念。
