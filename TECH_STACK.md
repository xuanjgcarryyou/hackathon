## Why This Stack

- **Python/FastAPI（後端）：** FastAPI 的 Pydantic v2 直接驗證 API 合約資料型別，async 支援良好，`anthropic` SDK 原生 Python 版本成熟。WEI 熟悉 Python。
- **Node.js/Express（中介層）：** Node.js 事件驅動模型天然適合 WebSocket 長連線管理；`ws` 套件輕量、無多餘依賴；作為 API Gateway 的轉發邏輯簡單，不需複雜框架。
- **React + Vite（前端）：** LUO 熟悉 React；Vite 啟動快速，開發體驗好；TailwindCSS 讓 UI 在 hackathon 時間內可以做得好看。
- **PostgreSQL：** B2B 資料（公司、訂單、容器批次）有明確關聯性，需要 ACID 保證；hackathon 常用，Docker image 穩定。
- **Docker Compose：** 全容器化確保 Demo 環境與開發環境一致；任何人的電腦都可以 `docker compose up` 跑起來，不依賴本地 Python/Node 版本。

## Decision

使用 Python/FastAPI（後端）+ Node.js/Express（中介層）+ React/Vite（前端）+ PostgreSQL（資料庫）全部以 Docker Compose 容器化部署。唯一外部依賴為 Claude API（`claude-sonnet-4-6`），用於 ESG 報表生成。開發時三人各自在獨立容器中工作，透過 ARCHITECTURE.md 定義的 8 個 REST 合約與 4 個 WebSocket 事件整合，中介層以 stub 回應先讓前端可以獨立開發。

---

## Rationale

- 選 FastAPI 而非 Django/Flask：Pydantic 合約驗證直接對應 ARCHITECTURE.md 的 Interface Contracts；async 支援 Claude API 非同步呼叫
- 選 Node.js 中介層而非 Python 處理 WebSocket：避免 WEI 同時維護 REST API 和 WebSocket，降低單點複雜度
- 選 PostgreSQL 而非 SQLite：B2B 場景有多表關聯查詢（公司 → 訂單 → 容器批次 → ESG 彙總），PostgreSQL 的 JOIN 效能與 ACID 保證更可靠

## Rejected Options

| 選項 | 拒絕原因 |
|------|---------|
| SQLite | 不支援高並發寫入（Demo 現場多人同時掃碼可能 lock）；容器化後 volume 管理麻煩 |
| Firebase / Supabase | 外部服務，Demo 時需要網路；違反「可離線 Demo」原則 |
| Next.js（全端） | 三人共用一個服務無法平行開發；SSR 在 hackathon 場景無必要 |
| GraphQL | 合約協調成本高，hackathon 時間緊；REST 合約更直觀 |
| Django REST Framework | 比 FastAPI 重，Pydantic v2 整合需要額外設定 |

## Known Constraints

- Claude API 需要網路連線：Demo 現場若無網路，ESG 生成失敗 → Fallback：展示預先生成的範例報告
- Docker Desktop 須在 Demo 前手動啟動：已記錄於 HACKATHON_BRIEF.md 風險表
- `CLAUDE_API_KEY` 由 WEI 持有並在開工時寫入 `.env`
